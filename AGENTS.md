# AGENTS.md

## 1. 프로젝트 개요
이 저장소의 목표는 **Python으로 동작하는 mini Redis 스타일의 TCP key-value 서버**를 구현하는 것이다.

이 프로젝트의 핵심은 다음 4가지를 직접 이해하고 구현하는 데 있다.
- TCP 서버-클라이언트 통신
- 명령어 파싱
- 메모리 기반 저장소
- 테스트 및 문서화

이 프로젝트는 **Redis 전체를 복제하는 것**이 목표가 아니다.  
오늘 기준 목표는 **작게 동작하는 MVP를 완성하는 것**이다.

---

## 2. 오늘 구현 범위(MVP)
### 필수 기능
- `PING`
- `SET key value`
- `GET key`
- `DEL key`

### 시간이 남으면 추가
- `EXPIRE key seconds`
- `TTL key`

### 오늘 하지 않는 것
- Flask / FastAPI
- Web UI
- persistence(파일 저장/복구)
- replication / clustering / scale-out
- list / set / hash / pub-sub
- 과도한 성능 최적화

---

## 3. 전체 구조
이 프로젝트는 아래 3개 레이어로 분리한다.

### `storage.py`
- 실제 데이터를 저장/조회/삭제하는 저장소 레이어
- 내부적으로 `dict` 사용
- TTL을 붙일 경우 만료시간도 관리
- 네트워크 코드 금지
- 파싱 코드 금지

### `parser.py`
- 클라이언트가 보낸 문자열 또는 RESP 입력을 해석하는 레이어
- `(command, args)` 형태로 반환
- 인자 개수 검증
- 데이터 저장 금지

### `server.py`
- TCP 소켓 서버 레이어
- 연결 수락, 데이터 수신, parser 호출, storage 호출, 응답 전송 담당
- 저장소 로직 재구현 금지
- 파싱 로직 재구현 금지

### `tests/`
- 기능 검증 레이어
- storage / parser / server 동작 테스트

## 4. 공통 인터페이스 규약
이 규약은 팀원 3명 모두 반드시 지켜야 한다.
규약을 바꾸면 테스트와 README도 같이 수정해야 한다.

### Storage 규약
- `set(key, value) -> "OK"`
- `get(key) -> value | None`
- `delete(key) -> bool`
- `expire(key, seconds) -> bool`  (선택)
- `ttl(key) -> int`  (선택)
  - `-2`: key 없음
  - `-1`: 만료시간 없음
  - `0 이상`: 남은 초

### Parser 규약
- `parse_command("PING") -> ("PING", [])`
- `parse_command("SET a 1") -> ("SET", ["a", "1"])`
- 잘못된 입력은 명확한 예외 또는 문서화된 에러 형태로 처리

### Server 응답 규약
- `PING` -> `PONG`
- `SET a 1` -> `OK`
- `GET a` -> 저장된 값
- 없는 key 조회 -> `nil`
- `DEL a` 성공 -> `1`
- `DEL a` 실패 -> `0`
- 잘못된 명령 -> `ERROR ...`

## [1번 팀원] 저장소 담당

### 역할
1번 팀원은 **데이터를 실제로 저장하고 관리하는 저장소 레이어**를 만든다.
이 사람이 만든 `storage.py`가 프로젝트의 기준이 된다.

### 담당 파일
- `storage.py`

### 반드시 구현할 것
- 내부 `dict` 기반 저장 구조
- `set(key, value)`
- `get(key)`
- `delete(key)`

### 시간이 남으면 구현
- `expire(key, seconds)`
- `ttl(key)`
- 만료된 key 자동 정리

### 책임
- 데이터가 정확히 저장되는지
- 조회가 정확히 되는지
- 삭제가 정확히 되는지
- TTL을 붙인다면 만료 시점 로직이 맞는지

### 구현 순서
1. `Storage` 클래스 만든다.
2. `self.data = {}` 구조를 만든다.
3. `set()` 구현
4. `get()` 구현
5. `delete()` 구현
6. 시간이 남으면 `self.expiry = {}` 추가
7. `expire()` 구현
8. `ttl()` 구현
9. 만료된 key를 `get()` 시 자동 삭제하도록 처리

### 상세 규칙
- `storage.py` 안에 socket 코드 넣지 말 것
- parser 코드 넣지 말 것
- print 디버깅 코드 남발하지 말 것
- 서버 응답 문자열(`PONG`, `ERROR` 등)은 storage에서 결정하지 말 것
- storage는 저장소 기능만 담당할 것

### 권장 반환값
- `set("a", "1")` -> `"OK"`
- `get("a")` -> `"1"`
- `get("없는키")` -> `None`
- `delete("a")` -> `True`
- `delete("없는키")` -> `False`

### TTL 추가 시 규칙
- 없는 key에 `expire()` -> `False`
- 만료시간 없는 key의 `ttl()` -> `-1`
- 존재하지 않는 key의 `ttl()` -> `-2`
- `set()`으로 값을 다시 덮어쓰면 기존 TTL은 제거하는 방식 권장

### 직접 확인할 것
- set 후 get이 되는지
- delete 후 get이 `None`인지
- 없는 key delete가 `False`인지
- 같은 key를 여러 번 set 하면 최신 값이 유지되는지
- TTL 구현 시 시간이 지나면 사라지는지

### 완료 조건
아래가 되면 1번 팀원 작업은 1차 완료다.
- `storage.py`가 독립적으로 실행/테스트 가능함
- `set/get/delete` 정상 동작
- 함수 반환값이 문서 규약과 일치
- 2번, 3번 팀원이 바로 사용할 수 있게 인터페이스 공유 완료

## [2번 팀원] 파서 + 서버 담당

### 역할
2번 팀원은 **클라이언트 입력을 해석하고, TCP 서버를 통해 storage와 연결하는 역할**을 담당한다.
쉽게 말하면, 사용자가 보낸 명령을 읽어서 1번 팀원의 storage에 연결해주는 사람이다.

### 담당 파일
- `parser.py`
- `server.py`

### 반드시 구현할 것
- 문자열 명령 파싱
- TCP socket 서버
- parser 결과를 storage 호출로 연결
- 응답 문자열 반환

### 시간이 남으면 구현
- RESP 파싱 지원
- 멀티 클라이언트 처리를 위한 threading

### 책임
- `SET a 1` 같은 입력을 정확히 해석하는 것
- 잘못된 명령을 걸러내는 것
- storage 결과를 규약에 맞는 문자열 응답으로 바꾸는 것
- 서버가 실제로 연결을 받고 응답을 돌려주는 것

### 구현 순서
1. `parser.py` 먼저 만든다.
2. `parse_command()` 함수를 만든다.
3. 공백 기준 문자열 분리 구현
4. 명령어 대문자 통일
5. 인자 개수 검증
6. 그 다음 `server.py` 구현
7. socket 생성 -> bind -> listen -> accept 흐름 구성
8. 받은 문자열을 parser에 전달
9. parser 결과를 storage 함수와 연결
10. 결과를 문자열로 응답

### parser 규칙
- `PING` -> 인자 0개
- `SET key value` -> 인자 2개
- `GET key` -> 인자 1개
- `DEL key` -> 인자 1개
- 잘못된 입력은 명확한 에러 처리

### server 규칙
- `PING` -> `PONG`
- `SET` -> storage.set 호출 후 `OK`
- `GET` -> 값 있으면 값, 없으면 `nil`
- `DEL` -> 삭제 성공 시 `1`, 실패 시 `0`
- 잘못된 명령 -> `ERROR ...`

### 상세 규칙
- `server.py` 안에 저장 로직 직접 구현 금지
- `server.py` 안에 파싱 로직 직접 구현 금지
- `parser.py`와 `server.py` 역할 섞지 말 것
- 오늘은 가장 단순한 구조를 우선할 것
- 먼저 단일 클라이언트 기준으로 완성하고, 시간 남으면 threading 추가

### RESP 관련 규칙(시간 남으면)
RESP를 붙일 경우 parser만 확장한다.
- `*` -> 배열
- `$` -> bulk string
- 클라이언트 명령은 RESP 배열로 해석
- 응답은 `+`, `:`, `$`, `-` 형식으로 직렬화 가능
- 단, storage 레이어는 RESP를 몰라도 되도록 유지

### 직접 확인할 것
- 서버가 포트에 정상적으로 뜨는지
- `PING` 왕복이 되는지
- `SET a 1` 후 `GET a`가 `1`인지
- `DEL a` 후 `GET a`가 `nil`인지
- 잘못된 입력에서 에러 응답이 나오는지

### 완료 조건
아래가 되면 2번 팀원 작업은 1차 완료다.
- `parser.py` 동작
- `server.py` 실행 가능
- TCP 연결 후 `PING/SET/GET/DEL` 왕복 가능
- 응답 형식이 문서 규약과 일치
- 3번 팀원이 통합 테스트를 작성할 수 있을 정도로 실행 방법 공유 완료

## 9. 수동 테스트 예시
최소한 아래 흐름은 직접 확인해야 한다.

```text
PING -> PONG
SET a 1 -> OK
GET a -> 1
DEL a -> 1
GET a -> nil
```

TTL까지 구현했다면 예시:

```text
SET a 1 -> OK
EXPIRE a 3 -> 1
GET a -> 1
(3초 후)
GET a -> nil
TTL a -> -2
```

## 12. 향후 확장 메모
오늘 MVP가 끝난 뒤 확장할 수 있는 항목은 아래와 같다.
- RESP 프로토콜 정식 지원
